## Testing 
- Always run a full build before running tests. Never use --no-build. If E2E or unit tests fail, verify the build succeeded first before diagnosing as flakiness or pre-existing failures.
- The stable regular test lane is `dotnet test src/Softwareschmiede.Tests/Softwareschmiede.Tests.csproj --filter "Category!=OsInterface"`. Run OS-interface tests separately with `--filter "Category=OsInterface"`; those include E2E, ConPTY, Clipboard, real process-start, and similar environment-sensitive tests.
- This project requires .NET Desktop workload to run tests. Verify the correct test configuration before running; do not repeatedly retry a config the user has already flagged as broken.
- When a WPF E2E test fails with "element not found" / timeout in `WpfTestBase.WaitForElement` or `WaitWhileMainHandleIsMissing`, do NOT default to "environment flakiness / no interactive desktop session" as the explanation. First check the launched app's own log file (`src/Softwareschmiede.App/bin/<Config>/<TargetFramework>/logs/softwareschmiede-*.log` of the test run's process) for a startup exception (e.g. `[ERR] MainWindow konnte nicht angezeigt werden.`). A crash during `App.xaml.cs` startup (e.g. XamlParseException from a bad resource path) looks identical from the test's perspective to a genuinely missing/slow window, but has a completely different, fixable root cause. Only fall back to the "no interactive session" explanation once the app log confirms the window/process actually started cleanly and no exception was thrown.
- **NEVER run `dotnet test` (or any Bash command containing that substring) with `run_in_background: true`.** Always run it synchronously/foreground and wait for it to finish. Reason: the `build_before_test.py` PreToolUse hook holds a cross-process build lock (`.claude/.locks/dotnet-build.lock`) for the entire duration of the test run — released only by the `release_build_lock.py` PostToolUse hook once the *actual* `dotnet test` Bash command completes. For a backgrounded call, the Bash tool call returns (and therefore PostToolUse fires) immediately, releasing the lock seconds into a multi-minute test run. The Stop hook (`test-csharp-startup.ps1`, fires whenever your own turn ends) then acquires the now-free lock and does a `dotnet build --no-incremental` + `dotnet run` + kill cycle on `Softwareschmiede.App.csproj` **while the E2E test run is still using that same executable**, corrupting `bin/` mid-run (missing `Softwareschmiede.App.runtimeconfig.json`, mass `TimeoutException`/"Desktop Runtime nicht gefunden" failures across nearly all E2E tests). This was reproduced live during the issue-114 investigation. A technical block (`sys.exit(2)` in the PreToolUse hook when `tool_input.run_in_background` is true) was attempted and did **not** prevent the Harness from still backgrounding the command — so this is a hard behavioral rule, not something the tooling currently enforces for you. If a sub-agent needs to run `dotnet test`, tell it explicitly not to background the call.
- **Always set `SOFTWARESCHMIEDE_SKIP_CONPTY_TESTS=1` when running `dotnet test` in this sandbox** (e.g. `SOFTWARESCHMIEDE_SKIP_CONPTY_TESTS=1 dotnet test ...` on bash, or `$env:SOFTWARESCHMIEDE_SKIP_CONPTY_TESTS='1'; dotnet test ...` on PowerShell — remember each Bash/PowerShell tool call starts a fresh shell, so this has to be set inline on every invocation, not once per session). Reason: 14 E2E tests start a CLI process via ConPTY (`Softwareschmiede.Tests/E2E/E2E_ConPty*.cs` and related). In this specific agent sandbox, the ConPTY child process is confirmed (via `KiAusfuehrungsService.TryGetExitCode`'s real exit code plus an empty `PseudoConsoleSession.Buffer` while the raw `cmd.exe` prompt leaks into the ambient test-runner console) to not get isolated to the pseudo-console, so it never receives the delayed command and the tests time out waiting for UI that never appears — this is confirmed **not** a code bug (production ConPTY code matches Microsoft's reference implementation) and confirmed to work fine from Visual Studio / a real interactive session. Two attempts at an automatic runtime probe (`ConPtyEnvironmentProbe`, tried once launching the ConPTY child directly from the test host, then once from a real `Softwareschmiede.App.exe` instance to match the production process tree) both produced **false negatives in Visual Studio** — i.e. incorrectly skipped tests that were actually passing there — so the mechanism was deliberately reverted to this explicit, agent-set environment variable instead of auto-detection. Without it, these 14 tests fail with an opaque `TimeoutException` in this sandbox every single run; with it, they report a clean, honest "Skipped" instead. Never set this variable when telling the user or a human to run tests themselves — it must only be used for automated runs from within this sandbox.
- **E2E tests using FlaUI must always be consolidated to the minimum number of test methods, without ever reducing the coverage of tested aspects.** Each FlaUI E2E test method starts a real app window and drives it via UI automation, so execution time — not method count or "one assertion per test" style — is the constraint that matters here. Combine related user-facing scenarios into a single test method by chaining steps against the same running app instance instead of relaunching per aspect. Example: creating, editing, and deleting a project can be one test method that walks through all three actions on the same instance; equally, a single method can create several projects and exercise different interactions across them. The goal is always to keep the total wall-clock runtime of the FlaUI E2E suite as short as possible while still fully covering every distinct behavior.

- Never trust sub-agent completion or test-pass reports at face value. Independently verify build cleanliness and test results (e.g., confirm files were actually created, code was actually removed) before reporting success to the user.

## Self-Hosting-Risiko: NIEMALS Softwareschmiede.App.exe beenden

Dieses Projekt wird mit sich selbst weiterentwickelt: Die laufende Claude-Code-Session (du) wird typischerweise **innerhalb** einer laufenden `Softwareschmiede.App.exe`-Instanz gehostet/gestartet (Desktop-Installation, z. B. `C:\Users\<user>\Desktop\Softwareschmiede\Softwareschmiede.App.exe`, nicht das Build-Ausgabeverzeichnis im Repo). Das bedeutet: **jede** laufende `Softwareschmiede.App.exe`, die du oder ein Sub-Agent auf dem System findest, könnte genau der Prozess sein, der dich gerade ausführt.

**Konkret bereits passiert:** Ein Sub-Agent stieß beim `dotnet build` auf einen Datei-Lock durch eine laufende `Softwareschmiede.App.exe` (MSB3027-Fehler beim Kopieren der DLL) und versuchte vermutlich, den blockierenden Prozess zu beenden, um den Lock aufzulösen. Dabei wurde die Instanz beendet, die die Haupt-Session hostete — die Session wurde dadurch unbewusst neu gestartet, der laufende Hintergrund-Task verwaist (`status: stopped`, Notification erst nach manueller Wiederaufnahme).

**Regeln:**
- Beende **niemals** eigenständig einen `Softwareschmiede.App.exe`-Prozess, auch nicht um einen Build-Lock aufzulösen — weder direkt noch über einen Sub-Agenten.
- Trifft ein `dotnet build` auf einen MSB3027/MSB3026-Kopierfehler wegen einer gesperrten `Softwareschmiede.App.exe`-DLL, informiere den Anwender und bitte ihn, die blockierende Instanz selbst zu schließen (er kann anhand von Fenstertitel/PID/Startzeit beurteilen, ob es eine unwichtige Test-Instanz oder die Host-Instanz ist — das kannst du von außen nicht sicher unterscheiden).
- Weiche stattdessen auf einen Teil-Build aus, der das App-Projekt nicht benötigt (z. B. `dotnet build src/Softwareschmiede/Softwareschmiede.csproj` für Domain-/Service-/Migrationscode), und dokumentiere transparent, dass die volle Verifikation (inkl. App-abhängiger Tests) noch aussteht.
- Verwaiste, aber eindeutig unabhängige Prozesse (z. B. `vstest.console` ohne Fenster) sind unkritisch und dürfen nach Rückfrage beendet werden — das Risiko betrifft ausschließlich `Softwareschmiede.App.exe`.
